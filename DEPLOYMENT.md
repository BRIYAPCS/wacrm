# Deploying wacrm

The complete, self-contained guide to standing up a **new wacrm instance
from zero** — one Supabase project + one app deployment per customer.

This repo is a **template you redeploy per customer**: each customer gets
their own Supabase project, their own domain, their own WhatsApp number,
and their own isolated data. Nothing is shared between instances. Once
you've done it once, a new customer is ~20 minutes of mostly clicking.

> **Doing this for another customer?** Jump to the
> [per-customer runbook](./docs/new-customer-checklist.md) — a tight
> copy-paste checklist that references the detail here.

---

## What you're deploying

| Layer | What runs it |
|---|---|
| **App** | Next.js 16 (App Router) — Hostinger Managed Node.js, Vercel, Railway, or any Node ≥ 20 host |
| **Database + Auth + Storage + Realtime** | One **Supabase** project (Postgres + RLS) |
| **WhatsApp** | Meta WhatsApp Cloud API (the customer's own number) |
| **Transactional email** | SMTP configured **inside the Supabase project** (powers team invites + password reset) |
| **Scheduled jobs** | GitHub Actions cron (host-agnostic) — or Vercel Cron / a server crontab |

The running app talks to Supabase over the API keys. The **only** thing
that needs a raw Postgres connection is applying migrations
(`npm run db:deploy`), which you run once per new migration.

---

## Prerequisites

- **Node.js ≥ 20** and `npm` on the machine you run setup from.
- A **Supabase** account (free tier is fine to start).
- A **Meta for Developers** account with a WhatsApp Business app + a phone
  number (can be added after the app is live — inbound webhooks are the
  only thing blocked without it).
- An **SMTP provider** (Resend, Postmark, SendGrid, SES…) — **required**,
  because sign-up is invite-only and invites are emailed. See
  [Step 4](#step-4--smtp--auth-urls-in-supabase-required).
- A **domain** for the customer (needed for the WhatsApp webhook, which
  requires HTTPS).

## Step 0 — Get the code

```bash
git clone https://github.com/<your-org>/wacrm.git customer-acme
cd customer-acme
npm install
```

Use a **separate clone/branch per customer** if you customise branding per
client, or a single repo you redeploy with per-customer env — either works.

---

## Step 1 — Create the Supabase project

1. [supabase.com](https://supabase.com) → **New project**. Pick a region
   close to the customer. Save the **database password** you set — you'll
   need it in Step 3.
2. **Project Settings → API** — copy these three:
   - `NEXT_PUBLIC_SUPABASE_URL`
   - `NEXT_PUBLIC_SUPABASE_ANON_KEY`
   - `SUPABASE_SERVICE_ROLE_KEY` (secret — server-only)
3. **Connect → Session pooler → URI** — copy the connection string and
   replace `[YOUR-PASSWORD]` with the DB password from step 1. This is
   `SUPABASE_DB_URL`.
   > Use the **Session pooler** (port `5432`), **not** the Transaction
   > pooler (`6543`) — the migrations run DDL that needs a session
   > connection.

> **Why Supabase specifically?** The schema depends on Supabase's `auth`,
> `storage`, and `realtime` schemas and on RLS driven by `auth.uid()`. A
> bare Postgres host (Neon/RDS) can't run it without a rewrite. Stay on
> Supabase.

---

## Step 2 — Configure environment

Run the assisted setup — it creates `.env` from the template, **generates
the two secrets** you'd otherwise fumble (`ENCRYPTION_KEY`,
`AUTOMATION_CRON_SECRET`), and tells you exactly which values are still
missing:

```bash
npm run setup
```

Then open `.env` and paste the four Supabase/Meta values. Re-run
`npm run setup` any time to re-check what's still missing.

See the full [environment variable reference](#environment-variable-reference)
below for every var. The minimum to boot:

```
NEXT_PUBLIC_SUPABASE_URL=…
NEXT_PUBLIC_SUPABASE_ANON_KEY=…
SUPABASE_SERVICE_ROLE_KEY=…
SUPABASE_DB_URL=…
ENCRYPTION_KEY=…            # generated by `npm run setup`
META_APP_SECRET=…           # can add later; only inbound webhooks need it
```

> **`ENCRYPTION_KEY` is load-bearing and permanent.** It encrypts every
> WhatsApp/AI token (AES-256-GCM). If you change it later, every stored
> token becomes undecryptable and users must re-save their settings.
> Generate it **once per customer** and keep it safe.

---

## Step 3 — Apply the database schema

```bash
npm run db:deploy
```

This applies every migration in `supabase/migrations/` in order — all
tables, RLS policies, the `avatars` / `chat-media` / `flow-media` storage
buckets, functions, triggers, and the realtime publication:

```
→ Connecting to database…
  ▶  001_initial_schema.sql … done
  …
  ▶  045_invite_only.sql … done
✓ Database up to date — 45 migration(s) applied, 0 already present.
```

It's **idempotent and safe to re-run** — applied versions are recorded in
`wacrm_schema_migrations` and skipped. When you pull new migrations later,
just run it again; only the new ones apply. More detail + troubleshooting:
[docs/database-setup.md](./docs/database-setup.md).

---

## Step 4 — SMTP + Auth URLs in Supabase (required)

wacrm is **invite-only**: the first user becomes the owner, and everyone
else joins by **email invitation**. That means Supabase must be able to
send email, and must trust your app's redirect URLs.

**4a. Custom SMTP** — Supabase Dashboard → **Project Settings →
Authentication → SMTP Settings** → enable Custom SMTP and fill in your
provider's host/port/username/password + a verified sender address. The
built-in Supabase mailer is dev-only (a few messages/hour, lands in spam)
— you **must** set this or invites won't deliver. Full walkthrough +
provider table: [docs/email-setup.md](./docs/email-setup.md).

**4b. Auth URLs** — Supabase Dashboard → **Authentication → URL
Configuration**:
- **Site URL** = your production domain (e.g. `https://crm.acme.com`).
- **Redirect URLs** — add `https://crm.acme.com/**` (a wildcard covers
  the invite accept page `/accept-invite` and password-reset
  `/reset-password`). Without this, the emailed links are rejected as
  untrusted redirects.

> Skipping SMTP is the #1 "invites don't arrive" cause. Send yourself a
> test invite (Step 8) before you hand the instance over.

---

## Step 5 — Deploy the app

Deploy anywhere Node ≥ 20 runs. Set **all** the environment variables from
your `.env` in the host's dashboard (never commit `.env`).

### Hostinger (recommended)
hPanel → **Websites → Create → Node.js**, connect your repo/branch, paste
the env vars, push to `main`. Free SSL + domain, managed Node, logs in the
panel. See the [README](./README.md#-deploy-on-hostinger-recommended).

### Vercel
Import the repo, add the env vars (Production), deploy. Add your domain.

### Any VPS / Node host
`npm run build && npm start` behind a reverse proxy that terminates TLS.
`npm start` serves on `:3000`.

**After it's live:** set `NEXT_PUBLIC_SITE_URL` to the real domain (used
for OG/sitemap and any link generated outside a request context), and make
sure the Supabase **Site URL / Redirect URLs** from Step 4b match.

## Step 6 — Connect WhatsApp (Meta Cloud API)

Inbound messages are the only thing gated on this — do it once the app is
live on its domain.

1. **Meta for Developers → your App → App Settings → Basic**: copy the
   **App Secret** → set `META_APP_SECRET` (verifies the HMAC on every
   inbound webhook). Optionally copy the **App ID** → `META_APP_ID` (only
   needed for image-header templates).
2. In the app: **Settings → WhatsApp** — enter the customer's
   `phone_number_id`, WABA id, a permanent **access token**, and a
   **verify token** (any random string). Save; the app registers the
   number with Meta.
3. In **Meta → WhatsApp → Configuration → Webhook**:
   - **Callback URL** = `https://crm.acme.com/api/whatsapp/webhook`
   - **Verify token** = the same string you entered in the app
   - Subscribe to the **messages** field.

Detail lives in the in-app Settings → WhatsApp screen, which walks the
token/registration state and surfaces Meta errors inline.

---

## Step 7 — Schedule the background jobs

Automations "Wait" steps, flow timeouts, and scheduled messages are
drained by three cron endpoints. A **committed GitHub Actions workflow**
(`.github/workflows/cron.yml`) pings all three every 5 minutes — no server
crontab needed. One-time setup (repo → **Settings → Secrets and variables
→ Actions**):

- **Variable** `BASE_URL` = `https://crm.acme.com` (no trailing slash)
- **Secret** `CRON_SECRET` = the same value as the app's
  `AUTOMATION_CRON_SECRET`

Until `BASE_URL` is set the job no-ops (never fails red). Prefer Vercel
Cron or a server crontab instead? Pick **one** to avoid double-firing —
options in [docs/automations-and-cron.md](./docs/automations-and-cron.md).

---

## Step 8 — Bootstrap the owner + invite the team

1. Visit `https://crm.acme.com` → **/signup is invite-only**, but the
   **very first sign-up on a fresh instance becomes the owner**. Create
   the owner account (this is you or the customer's admin).
2. **Settings → Team members → Invite** — enter a teammate's email + role
   (admin / agent / viewer). They get an email with a link to set a
   password on `/accept-invite` and land in the shared workspace.
3. Random emails **cannot** self-register — enforced in the database, not
   just the UI.

> **Escape hatch (no lockout):** if you ever need to temporarily re-open
> public sign-up, run
> `UPDATE app_settings SET public_signup_enabled = true;` in Supabase SQL
> and set `NEXT_PUBLIC_ALLOW_SIGNUP=true` to restore the form. Set both
> back to re-lock.

## Post-deploy verification

- [ ] `npm run db:deploy` shows all migrations applied.
- [ ] App loads at the domain over **HTTPS**; `/login` renders.
- [ ] First sign-up created an **owner** account.
- [ ] A test **invite email arrives** and `/accept-invite` completes.
- [ ] **Settings → WhatsApp** shows *Connected* and the registration probe
      is green.
- [ ] Send a WhatsApp message **to** the number → it appears in the inbox
      (webhook working).
- [ ] Reply from the inbox → the customer receives it.
- [ ] GitHub Actions **cron** run is green (Actions tab → "Scheduled
      cron" → Run workflow).

## Environment variable reference

Set these in your host's env (and in `.env` for local + `db:deploy`).

### Required

| Variable | Where to get it | Notes |
|---|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | Supabase → Settings → API | Public. |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Supabase → Settings → API | Public. |
| `SUPABASE_SERVICE_ROLE_KEY` | Supabase → Settings → API | **Secret.** Server-only (webhook, engines, invites, API-key auth). |
| `SUPABASE_DB_URL` | Supabase → Connect → **Session pooler** → URI | Used **only** by `npm run db:deploy`. Port `5432`. |
| `ENCRYPTION_KEY` | `npm run setup` (or `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`) | 64 hex chars. **Permanent per customer** — don't rotate. |
| `META_APP_SECRET` | Meta → App Settings → Basic | Verifies inbound webhook HMAC. Required for inbound WhatsApp. |

### Recommended

| Variable | Notes |
|---|---|
| `NEXT_PUBLIC_SITE_URL` | Your canonical domain (no trailing slash). Used for OG/sitemap + links generated outside a request. |
| `AUTOMATION_CRON_SECRET` | Shared secret for the cron endpoints. Generated by `npm run setup`. Mirror it into the GitHub Actions `CRON_SECRET` secret. |

### Optional

| Variable | Notes |
|---|---|
| `NEXT_PUBLIC_ALLOW_SIGNUP` | `true` re-opens the public sign-up form (default: invite-only). Must pair with the DB `app_settings.public_signup_enabled` flag. |
| `META_APP_ID` | Only for creating message templates with an **image header**. |
| `ALLOWED_INVITE_HOSTS` | Comma-separated hostname allow-list for request-derived URLs (bare-metal hardening). |
| `WHATSAPP_TEMPLATES_DRY_RUN` | `true` in CI/dev to exercise the template UI without a real WABA. |
| `AI_REQUEST_TIMEOUT_MS` | Per-call AI provider timeout (default `30000`). |
| `AI_CONTEXT_MESSAGE_LIMIT` | Recent messages sent to the model as context (default `20`). |

> **No global AI key.** The AI assistant is bring-your-own-key **per
> account** (Settings → AI), stored encrypted under `ENCRYPTION_KEY`.
> Nothing to set in the environment. Any provider works — OpenAI,
> Anthropic, Gemini, Azure, OpenRouter, Groq, DeepSeek, Mistral, xAI,
> GLM, or a self-hosted / OpenAI-compatible endpoint. A self-hosted
> endpoint (Ollama, LM Studio, vLLM…) must be reachable from where the
> app runs. See [docs/ai-assistant.md](./docs/ai-assistant.md).

## Updating an existing instance

```bash
git pull            # or redeploy the new commit
npm install
npm run db:deploy   # applies only new migrations (idempotent)
```

Then redeploy the app. Check `CHANGELOG.md` for any **"Migration
required"** or **"Setup required"** notes on the versions you're crossing.

## Troubleshooting

| Symptom | Fix |
|---|---|
| `SUPABASE_DB_URL is not set` / placeholder | Fill it in `.env` (Step 1.3). Session pooler, port `5432`. |
| `password authentication failed` | Wrong DB password in the URI. Reset under Supabase → Database → Settings. |
| `SASL` / TLS / timeout on `db:deploy` | You're on the Transaction pooler (`6543`). Use the Session pooler (`5432`). |
| Invite emails never arrive | Custom SMTP not configured in Supabase (Step 4a), or sender domain SPF/DKIM not propagated. |
| Invite link says "invalid/expired" | Supabase **Redirect URLs** don't include your domain (Step 4b), or the link was already used / revoked. |
| Inbound WhatsApp not appearing | `META_APP_SECRET` unset/wrong, webhook URL/verify-token mismatch, or the number isn't registered (Settings → WhatsApp probe). |
| "Sign-up is invite-only" when you *want* open signup | Set `NEXT_PUBLIC_ALLOW_SIGNUP=true` **and** `UPDATE app_settings SET public_signup_enabled = true;`. |
| Automations "Wait" / scheduled messages never fire | Cron not wired — set `BASE_URL` + `CRON_SECRET` in GitHub Actions (Step 7). |
